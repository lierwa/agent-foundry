# 路线图：Agent Foundry

## 总览

当前脚手架已经证明了 runtime 的基本形态：package 可以创建 plan、执行任务、在审批点暂停，并在一个简单的操作台中展示结果。下一阶段的目标不是“继续堆功能”，而是把这个脚手架升级成一个可持久、可验证、可操作的稳定基线，这样后面再加新 package 和新集成时就不用反复重做底层。

## 阶段列表

- [ ] **Phase 1：Runtime 持久化与验证基线**  
  把易失性的内存行为替换成可持久化实现，并建立真正可运行的验证基线。
- [ ] **Phase 2：操作体验与可观测性**  
  让 workbench 和 API 在任务执行过程中更容易查看、调试和信任。
- [ ] **Phase 3：Package 扩展契约**  
  在保持 perfume 示例可用的前提下，把 package 扩展路径做扎实。

## 阶段详情

### Phase 1：Runtime 持久化与验证基线
**目标：** 让 runtime 不再依赖纯内存状态，并为关键流程补上自动化测试。  
**依赖：** 无（第一阶段）  
**覆盖需求：** [PLAT-01, PLAT-02, PLAT-03, QUAL-01, QUAL-02, QUAL-03]  
**成功标准（必须成立）：**
  1. 在默认的本地持久化开发路径下，任务和 memory 状态在 API 重启后仍然存在。
  2. Runtime 核心流程有自动化测试覆盖，包括 planning、approval、execution、completion。
  3. 根目录验证命令能稳定执行这套测试基线。
**计划数：** 3

Plans:
- [x] 01-01：在现有 runtime interface 后面设计并接入持久化 adapter。
- [x] 01-02：为 runtime 生命周期和 API 行为补上测试。
- [x] 01-03：把持久化开发模式的本地配置与使用方法补进文档和脚本。

### Phase 2：操作体验与可观测性
**目标：** 在不改变 core package 契约的前提下，改善操作台体验和 runtime 可观测性。  
**依赖：** Phase 1  
**覆盖需求：** [WB-01, WB-02, WB-03, PLAT-04]  
**成功标准（必须成立）：**
  1. 操作人员无需翻原始 JSON，就能看到关键任务状态和审批上下文。
  2. API 和 workbench 失败时会给出清晰、可行动的提示。
  3. 日志和 trace 足够结构化，能帮助定位任务推进和审批中断问题。
**计划数：** 3

Plans:
- [ ] 02-01：改善任务列表和详情页的状态、审批、失败信息展示。
- [ ] 02-02：加强 API 边界层的错误处理和日志输出。
- [ ] 02-03：让 workbench 流程和新的可观测性、失败状态保持一致。

### Phase 3：Package 扩展契约
**目标：** 在保持示例 package 健康可用的前提下，让平台更容易接入新 package。  
**依赖：** Phase 2  
**覆盖需求：** [PKG-01, PKG-02]  
**成功标准（必须成立）：**
  1. 在完成持久化和可观测性改动后，perfume package 依然通过验证。
  2. 新增 package 时，在 shared schema、core 注册和 UI 暴露层有明确可遵循的模式。
  3. 整个仓库读起来更像一个可复用平台基线，而不是单一 demo。
**计划数：** 2

Plans:
- [ ] 03-01：审查并加固 perfume package，作为参考实现。
- [ ] 03-02：把 package 扩展流程文档化，并通过一个额外 package 或 stub 证明这条路径可行。
